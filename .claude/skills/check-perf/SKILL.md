# 高并发性能风险分析

检测代码在高并发场景下的潜在性能风险，输出分级报告和修复建议。

## 触发时机

- git commit 之前（通过 hook 自动触发）
- 手动执行 `/check-perf`

## 检测范围

| 风险类别 | 严重程度 | 检测内容 |
|---------|---------|---------|
| **线程池问题** | 🔴 高 | 线程泄漏、拒绝策略、队列溢出、配置不当 |
| **锁竞争/死锁** | 🔴 高 | 锁粒度过粗、嵌套锁、死锁风险模式 |
| **内存泄漏** | 🔴 高 | ThreadLocal 未清理、集合无限增长、资源未释放 |
| **同步阻塞** | 🟡 中 | 异步上下文中的阻塞调用、IO 阻塞 |
| **连接池问题** | 🟡 中 | 连接泄漏、超时配置、池大小不当 |
| **上下文传递** | 🟡 中 | 线程池上下文丢失、TTL 使用不当 |

## 执行步骤

### 1. 获取变更文件

```bash
git diff --name-only --cached
git diff --name-only HEAD~1  # 如果没有 staged 文件
```

筛选 `.java` 文件进行分析。

### 2. 静态分析（可选）

如果项目配置了 SpotBugs 或 PMD，运行静态分析：

```bash
# SpotBugs（如果配置）
mvn spotbugs:check -q 2>/dev/null || true

# PMD（如果配置）
mvn pmd:check -q 2>/dev/null || true
```

### 3. 代码模式扫描

对变更文件进行模式匹配，检测以下风险：

#### 3.1 线程池风险

读取 `references/thread-pool-patterns.md` 中的检测规则。

**关键检测点：**
- 线程池创建后未正确关闭
- 使用 `Executors` 工厂方法（可能 OOM）
- 拒绝策略配置不当
- 线程池监控缺失

#### 3.2 锁竞争风险

读取 `references/lock-patterns.md` 中的检测规则。

**关键检测点：**
- `synchronized` 块内执行 IO 操作
- 嵌套锁（不同顺序获取锁）
- 锁对象使用 String 或装箱类型
- `ReentrantLock` 未在 finally 中 unlock

#### 3.3 内存泄漏风险

读取 `references/memory-leak-patterns.md` 中的检测规则。

**关键检测点：**
- `ThreadLocal.set()` 后无 `remove()` 调用
- 静态集合无限增长（无清理机制）
- 监听器/回调未注销
- 连接/流未在 finally 中关闭

#### 3.4 同步阻塞风险

读取 `references/blocking-patterns.md` 中的检测规则。

**关键检测点：**
- 在 `@Async` 方法中使用阻塞 IO
- `CompletableFuture.get()` 无超时
- 响应式流中调用阻塞操作
- HTTP 调用无超时配置

#### 3.5 上下文传递问题

**关键检测点：**
- 线程池执行任务时未使用 TTL 包装
- `GrayContext` 在异步边界未传递
- `TaskDecorator` 未配置

### 4. 生成报告

按严重程度分级输出：

```markdown

## 严重程度判定标准

| 级别 | 判定标准 | 示例 |
|-----|---------|------|
| 🔴 高危 | 可能导致系统故障、数据丢失、服务不可用 | ThreadLocal 未清理导致内存泄漏、死锁风险 |
| 🟡 中危 | 高并发下性能显著下降 | 锁粒度过粗、同步阻塞调用 |
| 🟢 低危 | 优化建议，影响较小 | 可优化的代码结构、潜在改进点 |

## 与现有检查的集成

本检查在 `/check-code` 之后执行，作为额外的质量卡点：

```
git commit
    │
    ├── /check-code（测试、覆盖率）
    │
    ├── /check-perf（性能风险）
    │
    └── 执行提交
```

## 约束规则

- 高危问题必须修复，否则阻止提交
- 中危问题建议修复，可标记为 TODO
- 低危问题作为优化建议，不阻止提交
- 每个问题必须提供具体的修复代码示例

## 快速检查命令

```bash
# 仅检查特定文件
/check-perf path/to/File.java

# 检查最近 N 次提交
/check-perf --since HEAD~3

# 输出 JSON 格式
/check-perf --format json
```
